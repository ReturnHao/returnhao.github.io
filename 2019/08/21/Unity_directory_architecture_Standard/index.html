Unity目录架构规范

Posted on 2019-08-21 | Post modified: 2019-09-11 | In Standard , Unity

本规范仅针对Unity工程文件的目录架构进行规范化，本规范默认文件名采用Pascal命名方法进行命名，代码部分的规范请参照C#代码规范部分。本规范制定的主要目的是统一标准，完善团队协作、提高沟通效率。在使用过程中，如果大家还遇到本规范不能解决的问题，欢迎发邮件跟作者讨论。

Introduction

在探讨目录架构之前，首先我们需要了解Unity的Folder系统。在这之后，我们将探讨几种可行的目录架构思路，并从中分析出一个相对完美的方案。最后是一些细节的界定和补充。

Unity Folder System

Special Folder Name System

本部分主要参照Special Name编写。

通常，你可以为你创建的文件夹选择任何你喜欢的名称。但是，有许多文件夹名称，Unity将其解释为一条指令，即应以特殊方式处理文件夹的内容。

例如，你必须将所有的Editor Scripts放在一个叫Editor的文件夹里，才能确保你所想要的效果。

Unity一共保留了8个不同的项目文件夹名称以指示其中的内容具有特殊用途，它们分别是：

  • Assets
  • Editor
  • Editor default resources
  • Gizmos
  • Plugins
  • Resources
  • Standard Assets
  • StreamingAssets

在此，我们选取几个常用的文件夹名进行重点说明。

Assets

主文件夹，包含所有工程需要用到的资源。

Editor

放置在名为Editor的文件夹中的脚本被视为编辑器脚本而不是运行时脚本。

Plugins

所有的扩展插件资源，一般为用C/C++编写的native DLLs，用以访问第三方代码库，调用系统资源或其他Unity内置功能。

Resources

一般用来存放需要通过代码初始化/使用的资源。通常被用于按需加载。

StreamingAssets

将文件放在名为StreamingAssets的文件夹中，这样它就可以不加改变地复制到目标平台，然后可以从特定文件夹中获取。

关于的StreamingAssets的应用及其扩展可以参照Streaming Assets，在此就不赘述了。

Associating with General Folder Name System

在进入下一部分之前，我们需要明确每一个特殊文件夹的性质。其中最令我们关心的，是文件系统的灵活性。

通常情况下，和现有的文件系统一样，同一个名称的文件夹可以被建立多次，只要它们分属在不同的父文件夹下，例如:

然而，为了保证访问的有效性，Unity对部分特殊文件夹进行了目录规定：

Name Quantity Place
Editor multiple anywhere inside the Assets folder
Editor default resources only one root of the Project
Gizmos only one root of the Project
Plugins only one root of the Project
Resources multiple anywhere inside the Assets folder
Standard Assets only one root of the Project
StreamingAssets only one root of the Project

可见，除了Editor和Resources之外，其他特殊文件夹均只能存放于根目录。

Directory architecture specification

Maintainability

在探讨目录架构规范之前，我们应该先了解一下可维护性。

可维护性是所有软件都应该具备的基本特点，对于任意一个Project，我们需要很快的知道其模块化、测试性、灵活性和可移植性这四个可维护性的基本组成部分的情况，并对软件进行相应的调整。

Unity Project’s maintainability

Unity在一个标准的Project中，总是会将资源(Assests)和调用(Scenes)分离，不同的Scene用来表示不同的项目场景，它们可以调用同一个位于Assests中的资源。

可维护性在Unity的直观体现十分明显：包(Packages)的可维护性。开发过程中经常需要进行包的导入和导出，如何快速的导入/导出这个场景，以及这个场景所需要的所有资源，同时不破坏这个大型项目的封装性，这是我们需要探究的。

Some Attempts

熟悉Unity的朋友可能知道，为了更好的整理文件调用资源，一般在Assets都包含Prefabs, Materials, Scripts等一些以Unity Standard Object的名字命名的文件夹，显然，在不存在复用（reuse）的情况下，这种命名可以说是相对最规范的一种了。

但当整个工程由于不得不复用某些组件而必须整合在一起的时候，这种命名方式就存在缺陷了，每个文件都被乱序的存放在标准命名的文件夹下，同时由于Unity糟糕的创建系统（默认New Scripts的创建路径为Assets/New Scripts.cs)，导致整个工程的维护性很差。

此时，重新以一种规范整合目录架构就变得亟须了。这里有两种常见的方法：

第一种方式并不改变Assets的一级结构，而是在一级结构里面建立各个项目的子文件夹，在子文件夹中分别存放这些资源。
第二种方法将每一个资源都统一成一个文件集合，并在文件集合中再对其进行分类存放。

然而，这两种方法都存在缺陷，第一种保证了Assets调用的逻辑（就像垃圾分类），并且可以很好的导入并不破坏目标文件夹的复用性，但同时破坏了其封装性。而第二种，完美的表示了每一个项目的封装性，可以快速导出资源并不破坏目标文件夹的可维护性，但同时破坏了其复用性以及导致了一些复用资源的空间浪费问题。

Finale

让我们回归Unity文件系统本身，重新回顾第一章发现，由于存在那些只能在特定位置建立的特殊文件夹，完全参照任何一种方案是不可能的。

那么，存不存在一种合理的方式，可以最大限度的保证其封装性，又能实现其复用可行性呢？答案很显然。

Some Details

§3.1. Public Resources

公共资源，顾名思义，是那些有多个项目用到的资源。公共资源在定义上更接近于C++中的基类，即任何非私有化的资源都应放在公共资源中。

同时公共资源也相当于一个接口，该接口对美工开放，所有所需要的图片等资源均应存放于Public中。

§3.2. Input Package

很多时候我们需要导入一些外部资源包，这些资源包跟项目一样，应该独立。

§3.3. Plural form and Others

在文件夹的命名中，我们时常会碰到是否复数化的问题，例如三级文件夹中的：Prefab和Prefabs，Scene和Scenes等。针对这类问题，我们统一所有的一，二，三级文件夹必须以复数化的形式存在，在三级文件夹之后，不做明确的文件夹命名规范。

同时，在导入导出的过程中，Unity会默认完全忽略以下命名方式的文件/文件夹：

  • 隐藏文件/文件夹
  • 以’.'为开始字符的文件/文件夹
  • 以’~'为开始字符的文件/文件夹
  • 以’cvs’命名的文件/文件夹
  • 扩展名为’.tmp’的的文件

请务必留意。

至此，我们完成了目录架构的规范化。
